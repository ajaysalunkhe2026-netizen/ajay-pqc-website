SentinelEdge V2.0: Technical Breakdown

By Ajay Salunke | Security Engineering Update

Hey everyone, I want to share a detailed technical breakdown of my recent security project: SentinelEdge V2.0.

When I built the first version, it was a simple script. It got the job done, but it had a critical flaw—it relied on basic HTTP redirects to block malicious traffic. The reality is that any standard automated scanner or experienced attacker would immediately see a "302 Redirect" and realize they hit a trap.

To fix this, I completely tore down the old code and rebuilt the system from scratch. Here is a deep dive into how the new version handles threats:

1. Multi-Modular Architecture

I moved away from a single-file script and split the system into three distinct, maintainable components:

  • The Real Application: The actual, legitimate site that normal users interact with.
  • The Honeypot (Dummy Site): A deliberately vulnerable-looking fake environment to keep attackers occupied.
  • The Defensive Gateway: The core engine that sits in front of both, managing and inspecting all incoming traffic.

2. Transparent Proxy Deception (The Core Upgrade)

This is the biggest improvement. The gateway now actively inspects incoming traffic for threats. If it detects an attack, it does not redirect the user.

Instead, it acts as a transparent proxy. It silently fetches the honeypot environment and serves it back to the attacker. The URL in their browser never changes, meaning the attacker thinks they are successfully exploiting the real server, while they are actually completely isolated in a fake trap.

3. Forensic JSON Logging

Defensive security is incomplete without logs. Every time the gateway intercepts an attack, it systematically records the exact timestamp, the payload used, and the routing details in a JSON audit file.

Building this was a fantastic hands-on experience in traffic routing, proxy mechanics, and defensive cybersecurity.
